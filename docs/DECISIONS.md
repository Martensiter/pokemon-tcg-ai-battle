# 意思決定ログ & 振り返り（DECISIONS）

**目的**：このプロジェクトの「**なぜそう決めたか**」「**何を間違えて直したか**」「**まだ怪しい判断**」を残し、
後から振り返って強化できるようにする。コードのdiffには"理由"も"誤り"も残らないので、ここに書く。

- `HANDOFF.md` = 運用how-to（何をすればいいか）
- `DECISIONS.md` = 判断の記録（**why** と **反省**）← 本書
- `HANDOFF_MACBOOK.md` = engine機の実測記録

> 新しい判断・修正をしたら、§2（ミス）か§3（要再検証）に1行足すこと。これが"強化"の入口。

## 0. 自動化・運用コンテキスト（セッションの頭の中＝コードに出ない前提）

- **2段構成**：収集 = ARM Hub（aarch64・numpyのみ・常駐）／検証・提出 = エンジンが載る実機（x86_64 or Apple Silicon native）。
  理由：エンジンはPokemon配布のネイティブ共有ライブラリで、Hub(aarch64)では `dlopen` 不可。
- **Hub常駐先 = `/home/node/.openclaw` だけが永続**。コンテナは揮発するのでここに repo/venv/data/manifest/cron を置く。
- **自動化 = OpenClaw cron**：30分毎 collect ＋ 4:00 UTC 日次 retrain→publish。
  **単一publisher**（collectorは local sink、`daily_pipeline` だけが Dataset version を切る）でレース回避。
- **提出は人間ゲート**：`competitions submit` が OAuth必須で無人化不可＋回帰を自動で出したくない。
- **監視 = Dataset の "updated" 日付を見るだけ**（専用cron/Actionsは不要。Hub全損まで自動検知したい時だけ `status.json` の `ts`）。
- **改善ループの正準形**：collect → offline retrain → verify → resubmit（**オンライン学習ではない**）。

## 3. まだ怪しい判断（要再検証＝次の強化候補）

- **value net が提出サンドボックスで実際にロードされているか未確認**。未ロードならヒューリスティックのみ＝弱い。
  → 既存 sims ログに `vnet=on/off` を足して次の提出で実証する（未実装・低コスト）。
- **567.5 < 1127.8 の解釈**：おそらく「土俵が変わった＋小サンプル」で**情報量低下ではない**（提出agentはfork元と同一・挙動改変なし）。
  ただし上の vnet 実証が済むまでは仮説。
- **policy統合方式（D5）**：root PUCT prior を選んだが、rollout方策／直接模倣の方が強いかは未検証。
- **抽出が MAIN・単一選択のみ**：非MAIN（ターゲット/ディスカード等）と複数選択を全捨て。
  カバレッジ損。`skip_*` カウンタで実量を見てから取り込み判断（follow-up）。
- **学習特徴が意味を持たない（D3）**：value=粗い盤面量、policy=不透明ID。相性/特性は**ヒューリスティック側だけ**が見る。
  layer B（エンジン由来の damage/type/weakness 特徴）で学習側にも相性を入れられる（提出はエンジンがあるので可、収集側は engine 機限定の抽出分岐になる）。
- **蒸留の教師＝勝者の手**：レーティングで重み付け（より強い相手の手を優先）した方が良いかは未検証。
